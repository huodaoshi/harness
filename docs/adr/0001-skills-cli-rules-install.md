# skills CLI：规则安装扩展（独立锁、分端目录、MVP 范围）

扩展 `cli/` 中的 `skills` 工具，使其除技能外还能从 Git 包安装 **Cursor / Claude Code** 用的项目规则；命令形态为 `skills rules add` 与基于锁文件的恢复。本文记录三项易被误读的结构选择。

**状态：** 已接受

---

## 为何使用独立锁文件 `rules-lock.json`（不与 `skills-lock.json` 合并）

- **变更节奏不同**：技能包与规则包常来自不同上游、不同发版周期；绑进同一锁会导致「只为更新规则却动到技能条目」或反之，审查与回滚语义混杂。
- **恢复语义单一**：`experimental_install` 类命令若同时恢复两类资产，失败时难判断是技能源还是规则源问题；**分开锁**则技能与规则各有一条清晰、可单独重试的恢复路径。
- **团队协作更灵活**：可只提交 `rules-lock.json` 统一编辑器规则，而无须牵动技能锁；仓库也可阶段性只引入其一。

曾考虑的替代是将 `rules` 段嵌在 `skills-lock.json`。**未采纳**：单文件双领域会拉长字段路径、混淆 CI 缓存键与文档心智，收益不足以抵消合并成本。

---

## 为何采用仓库内分端目录（`rules/cursor/` 与 `rules/claude/`）

- **后缀与工具约定不同**：Cursor 规则常用 **`.mdc`**，Claude Code 的 `.claude/rules` 以 **`.md`** 为主；两端若共用同一目录，必须在安装时 **静默改扩展名**，易与包维护者本地文件不一致，也难以 diff。
- **显式镜像**：分端目录让「同一策略两套文件」成为**可见**选择；harness 等仓库可刻意对齐内容、分别保留正确后缀，由包发布方控制，而非 CLI 猜测转换规则。
- **安装逻辑简单**：`-a cursor` 只挂载 `rules/cursor/` 子树，`-a claude-code` 只挂载 `rules/claude/`，与 Agent 枚举一一对应，减少「同一文件服务两台编辑器」时的边界情况。

曾考虑的替代是单一 `rules/` + 目标端自动改后缀。**未采纳**：自动改名在 Windows 大小写、符号链接与可审计性上都会引入额外规则与故障面。

---

## 为何 MVP 不包含 `rules list` / `rules remove`

- **先打通主路径**：价值闭环是 **能从标准包安装到正确目录** + **`rules-lock.json` 可复现**；没有 list/remove 已能团队内落地与 CI 恢复。
- **列与删与 skills 同类复杂度**：需约定「已安装」的判定（符号链接目标、`--copy` 与目录覆盖、部分文件冲突），并与未来多包并存策略一致；与 add/lock **增量开发**风险更低。
- **避免双套半成品 UX**：list/remove 若设计与 `skills list/remove` 不完全对齐，会制造两套文档；待 `rules add` 与锁文件稳定后，再复用 installer 元数据做 list/remove，可减少返工。

MVP 保留 **单次只针对一个 Agent**、通过**执行两次**覆盖双端；list/remove 列入后续迭代。
